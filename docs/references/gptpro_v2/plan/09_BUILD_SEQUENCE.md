# Build Sequence

Stage 1 is the ingestion and provenance MVP. It accepts the packet zip, extracts filing markdown files, creates `CleanFiling`, section index, page-marker index, paragraph index, source spans, and document hashes. It produces no bidder rows. Acceptance is that all four reference filings ingest deterministically, preserve page markers, identify the Background and Reasons sections, and write paragraph evidence spans that later objects can cite.

Stage 2 is the minimal canonical skeleton. It creates `deals`, `process_cycles`, `actors`, `events`, `event_actor_links`, `judgments`, and `source_spans` tables with validation for evidence and referential integrity. Populate only manually or rules-based obvious facts at first: target, named advisors, named counsel, board committee names, signing events, and cycle shells. Acceptance is a runnable canonical store for all four deals with no floating objects and a review table export.

Stage 3 is proposal and bid extraction for simple cases. Implement actor alias extraction, dated proposal extraction, bid normalization for per-share cash values and ranges, and event-actor linking. Target PetSmart and Saks first because their bidder groups and bid values are comparatively explicit. Acceptance is that PetSmart Buyer Group, Bidder 2, and Bidder 3 bids and Saks Hudson’s Bay and Sponsor group bids match the test plan, with source spans and bid bounds.

Stage 4 is cycle and boundary handling. Implement process-cycle segmentation, formal-boundary judgments, boundary-absence markers, and cycle-regime judgments. Acceptance is that PetSmart has the November 3 final-round boundary, Saks has the July 2 formal-process candidate, Zep 2014 has null formal boundary with a termination endpoint, and Zep 2015 is a separate cycle from the 2014 process.

Stage 5 is dropout and scope judgments. Implement rejection, voluntary withdrawal, ambiguity, admission, valuation-comparability, and scope-validity judgments. Acceptance is that PetSmart eliminated parties are target-rejected when evidenced, Zep Party X is voluntarily withdrawn, Zep 2014 nonresponses are not converted into target screening, Providence Party D and Party E reengagement proposals are preserved without automatic admission, and Company H is out-of-scope for estimator bids.

Stage 6 is aggregate-count support. Implement anonymous aggregate actors, aggregate participation evidence, and cycle visibility flags. Acceptance is that Providence and Worcester preserves nine IOIs and the 11 strategic/18 financial outreach counts, Zep preserves 50 contacted and 25 confidentiality agreements, and Saks preserves go-shop counts even where individual parties are unnamed. Rows with hidden individual bid values must be marked incomplete rather than filled with invented values.

Stage 7 is auxiliary data retention. Implement advisor engagements, counsel engagements, board committees, deal terms, group memberships, prior relationships, go-shop windows, termination fees, reverse termination fees, voting agreements, and financing-condition terms. Acceptance is that all auxiliary facts listed in the reference test plan are queryable from canonical tables without reading the filings again.

Stage 8 is deterministic view derivation. Wire the canonical store to the `derive_views.py` projection and add the companion cycle-statistics projection for `J`, `g^S`, `r_23`, `mean_w`, and `S`. Acceptance is that every exported bidder row has all fields in the view schema, even when some values are null, and every null has a scope or comparability explanation.

Stage 9 is validation and audit hardening. Add blocking validation, soft flags, ambiguity queue, review master, cycle summary, bidder summary, deal index, and run memo. Acceptance is that all four filings export with no hard failures, while known ambiguities appear in the ambiguity queue.

Stage 10 is corpus-readiness. Run the pipeline on the four references as regression fixtures, then add new filings one at a time. A new filing is accepted only when it produces canonical records, evidence-backed judgments, and view rows without changing outputs for the four references unless a versioned rule change is intentionally made.

Stage 11 is reviewer workflow integration. Add a small review protocol for ambiguity-queue items: accept, override with evidence, defer, or mark unresolvable. Reviewer overrides must create new judgment records rather than editing source events. Acceptance is that a reviewer can resolve a low-confidence dropout label and the bidder-row projection changes only through the revised judgment.
