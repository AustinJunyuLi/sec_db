# SEC Merger Filing Pipeline Plan

This packet is an implementation plan for a filing-to-data pipeline that turns merger proxy narratives into a canonical structured deal representation and then into the bidder-cycle view consumed by the existing structural estimator. The core design choice is to avoid treating the estimator row as the data model. The estimator row is a deterministic projection from a richer representation containing deals, process cycles, participants, events, links, and evidence-backed judgments. That richer representation preserves adjacent material such as counsel, advisors, committees, go-shop terms, termination fees, prior relationships, diligence access, and process restarts, so later research views can be written without re-extracting the filings.

The plan is organized as follows. `01_RESEARCH_TARGET.md` restates the output contract, including how `derive_views.py` maps canonical tables into `estimation_bidder_rows` and cycle summaries. `02_DATA_MODEL.md` defines the intermediate and final schemas, including JSON sketches and type vocabularies. `03_ARCHITECTURE.md` gives the system graph and data flow. `04_INGESTION.md`, `05_EXTRACTION.md`, and `06_AGGREGATION.md` specify the three active construction layers: text normalization, evidence-backed extraction, and deterministic projection to bidder-cycle rows. `07_VALIDATION_AND_AUDIT.md` defines invariants, flags, evidence binding, and reproducibility. `08_TEST_PLAN.md` gives acceptance criteria against the four reference filings. `09_BUILD_SEQUENCE.md` stages the MVP into runnable releases. `10_DECISIONS.md` records architectural alternatives and chosen approaches. `11_OPEN_QUESTIONS.md` lists genuine unknowns and fields that cannot be reliably extracted from the trimmed filings alone.

The design is intentionally provider-neutral. It does not bind the system to any LLM vendor, API protocol, rate limit, streaming method, or cost model. It also does not alter the estimator, the model class, or the economic interpretation of the structural parameters. The estimator is treated as an existing consumer whose present view is defined by `derive_views.py`.

The four filings are examples, not a golden key. The plan therefore emphasizes general rules that are testable on those filings: explicit cycle segmentation, nullable formal boundaries, separate treatment of proposal events and judgments, support for anonymized and aggregate bidders, and dropout labels that distinguish target rejection from voluntary withdrawal when the text supports that distinction. Where the filing text is inherently ambiguous, the pipeline must preserve ambiguity rather than manufacture precision.

The most important implementation principle is evidence first. Every deal, cycle, actor, event, link, term, and judgment must point back to filing text through a source span or evidence bundle. A value without source text is not canonical data. Derived rows may contain nulls, low-confidence labels, or scope flags; they may not contain unsupported facts. This is especially important for formal-boundary detection and dropout-vs-screening classification, where silent overconfidence would directly contaminate the estimator.

The final deliverable of the implemented system is a canonical store plus deterministic views. The canonical store is stable enough to support future projections; the views are replaceable functions over that store. The bidder-cycle projection must reproduce every field expected by `derive_views.py`, while auxiliary views can later answer questions about go-shops, counsel, committees, breakup fees, advisor changes, prior bidder-target relationships, and similar features.
