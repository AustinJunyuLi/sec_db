# Architecture

The system is a layered construction pipeline. Each layer has a narrow output contract, writes durable artifacts, and never destroys evidence from the previous layer. The component graph is: input filing archive → document ingestion → source-map store → candidate extraction → candidate reconciliation → canonical store → validation and audit → deterministic view derivation.

The input layer accepts a zip of filing markdown files and optional filing metadata. Metadata may include EDGAR accession, filing URL, form type, filing date, and company identifiers. The markdown text remains the primary source for research facts. Metadata fills only fields that the trimmed narrative cannot contain, such as filing URL or form type.

The ingestion component normalizes text while preserving offsets. It removes operational noise such as repeated table-of-contents lines only after recording the original character positions. It indexes page markers, sections, and paragraphs. Its output is `CleanFiling` plus `SourceSpan` candidates for every paragraph. This layer does not extract bidders or bids; it only creates a reliable map from normalized text back to the filing.

The extraction layer runs multiple evidence-focused passes over the clean text. One pass identifies actors and aliases. One pass extracts dated process events. One pass extracts proposals and bid values. One pass extracts terms, advisors, counsel, committees, and financing or go-shop details. One pass classifies cycle boundaries, withdrawals, rejections, and formal-boundary candidates. The layer writes `ExtractionCandidate` objects, not final facts. Candidate objects may overlap or conflict, and every candidate has evidence spans.

The reconciliation layer converts candidates into canonical records. It merges aliases, assigns actor IDs, creates cycle IDs, links actors to events, normalizes bids, and emits judgments. This layer owns hard architectural decisions such as whether an aggregate bidder count creates anonymous bidder actors, whether an unsolicited letter is in scope, and whether a post-rejection proposal is estimator-eligible. It is deterministic given the candidate set and reconciliation configuration.

The canonical store is the single source of truth. It contains the tables required by `derive_views.py` and auxiliary tables for future research. The required tables are `deals`, `process_cycles`, `actors`, `events`, `event_actor_links`, and `judgments`. Auxiliary tables include advisor engagements, counsel engagements, board committees, deal terms, group memberships, prior relationships, and evidence spans. The canonical store is intentionally richer than the current estimator view.

The validation layer runs before any view is exported. It checks referential integrity, date consistency, evidence binding, bid bounds, required judgments, scope flags, and cycle segmentation. It separates hard failures from soft flags. A hard failure blocks export; a soft flag is written to the ambiguity queue but still permits export if the row is accurately marked.

The view derivation layer is pure projection. It does not call extraction and does not make new factual judgments. It runs the `derive_views.py` logic for `estimation_bidder_rows`, `estimation_auctions`, review tables, ambiguity queue, deal index, bidder summary, cycle summary, and run memo. Cycle-level statistics such as `J`, `g^S`, `r_23`, `mean_w`, and `S` are computed by an adjacent projection over the bidder rows and cycle judgments. The projection uses explicit inclusion rules, such as whether to count rows whose `scope_validity` is not estimation-ready.

The audit surface is built into the architecture. Analysts can inspect a bidder row, follow `bI_event_id` and `boundary_event_id` to canonical events, follow those events to source spans, and read the exact filing text. They can inspect a dropout judgment, see the evidence supporting the chosen value, and see any alternative value. They can also inspect aggregate-count rows and determine whether the filing hid individual bid values.

The architecture treats no-boundary cycles as ordinary records. Reconciliation creates a `formal_boundary` judgment for every cycle. If the filing supports a formal boundary, the judgment links to the transition event. If no boundary exists, the judgment value is null and links to a boundary-absence marker event. That marker supplies a date for deterministic ranking of informal proposals while preserving the null boundary value.

The architecture is batch-oriented and reproducible. A run takes a finite set of filings, writes canonical artifacts, validates them, and exports views. It does not rely on background work, live external lookups, or provider-specific behavior. Future provider binding can replace extraction engines as long as they emit the same candidate contracts.

Each component exposes explicit inputs and outputs. Ingestion receives raw filing text and produces clean text artifacts. Extraction receives only clean text plus source spans and produces candidates. Reconciliation receives candidates and produces canonical tables. Validation receives canonical tables and produces pass/fail reports plus ambiguity artifacts. Projection receives only validated canonical tables. This separation lets implementers replace one extraction method without rewriting cycle aggregation, validation, or view derivation.

Storage should be append-auditable even if implemented in a simple local database during MVP. Candidate artifacts, canonical records, validation reports, and exported views should be written under a run directory keyed by input hash and rule versions. The latest accepted run can be symlinked or indexed, but older accepted runs should remain available for regression comparison.
