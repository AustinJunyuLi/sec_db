# Validation and Audit

Validation protects the estimator from unsupported rows and protects analysts from opaque judgment calls. The validator runs after canonical reconciliation and before view derivation. It has hard checks that block export and soft checks that permit export but require review.

The first hard invariant is evidence binding. Every canonical deal, cycle, actor, event, event-actor link, judgment, advisor engagement, counsel engagement, board committee, deal term, group membership, and prior relationship must reference at least one `evidence_id`. The referenced source span must exist, must belong to the same filing, and must contain the quoted source text or a hash-matched excerpt. Derived values may cite the evidence of their source event or judgment, but no canonical object may float without provenance.

The second hard invariant is referential integrity. Every `cycle_id` in `events` must exist in `process_cycles`. Every `actor_id` in `event_actor_links` must exist in `actors`. Every judgment scoped to an actor, event, or cycle must point to an existing object. Every bidder actor included in the estimator view must have at least one event link in that cycle. Group memberships must reference valid grouped and member actors.

The third hard invariant is date consistency. Event date intervals must be valid. Exact dates must use full ISO dates. Rough dates must have normalized interval endpoints and a non-exact `date_precision`. A cycle start cannot be after its end. A real formal-boundary event must fall inside its cycle. A boundary-absence marker must fall at the cycle endpoint or at the latest supported process event. Post-signing go-shop events must not be placed inside a pre-signing primary cycle unless `cycle_relation` says the go-shop is a tail.

The fourth hard invariant is bid consistency. Numeric lower bounds cannot exceed upper bounds. `bid_value` must lie within its bounds unless the raw note explains a verbal ceiling or floor. Numeric bid events must have a unit, currency when relevant, and consideration type when the filing states one. Ranges used for `w_logwidth` must have positive lower and upper values. Bid values derived from aggregate ranges cannot be assigned to individual anonymous bidders unless the filing gives individual values.

The fifth hard invariant is required judgments. Every cycle must have `formal_boundary`, `cycle_regime`, and `cycle_visibility` judgments. Every bidder-cycle row must have `scope_validity`, `dropout_mechanism`, and, if it has an informal bid, `valuation_comparability`. Every non-null `formal_boundary` must have confidence. Every null `formal_boundary` must have a `no_formal_boundary_reason` or equivalent cycle-regime basis.

Soft flags identify rows needing analyst attention. A boundary is flagged when competing candidate dates exist, when the transition is inferred from process actions rather than explicit final-round language, or when the formal stage is weak. A bidder is flagged when its type is unknown, it is an anonymous aggregate actor, it belongs to a changing consortium, or its alias could refer to multiple entities. A bid is flagged when it is a verbal floor or ceiling, aggregate-only, non-cash, mixed consideration, enterprise-value rather than per-share, or missing a center. A dropout judgment is flagged when the value is `ambiguous`, low confidence, or has an alternative value.

The audit layer writes standard review artifacts. `review_master.csv` contains one row per event-actor observation with date, event type, bid fields, source text, page hint, and raw note. `ambiguity_queue.csv` contains all low-confidence judgments and all judgments with alternatives. `deal_index.csv` lists deal-level metadata. `bidder_summary.csv` lists bidder actors, subtypes, group status, and scope validity. `cycle_summary.csv` lists cycles, regimes, visibility, relationships, and formal boundaries. `run_memo.md` summarizes included deals and ambiguity counts.

Auditability must work at row level. A reviewer inspecting a PetSmart Buyer Group row can trace `bI` to the October 30 indication range, `bF` to the December 12 best-and-final offer, the boundary to the November 3 final-round decision, and the bidder type to group evidence. A reviewer inspecting a Zep 2014 row can see that process termination and voluntary-withdrawal language support no formal-stage screening. A reviewer inspecting Saks Company H can see why the letter is stored but excluded from estimator-eligible proposal events.

Reproducibility requires run metadata. Each run stores input file hashes, ingestion parser version, extraction configuration, reconciliation configuration, validation version, projection version, and timestamp. The system records candidate IDs and canonical IDs deterministically. Re-running the same versions on the same inputs should produce identical canonical records and views except for timestamped run memos.

Validation reports are not cosmetic. Hard failures must identify the object, failed invariant, evidence, and repair class. Soft flags must identify why the row is still exported. Analysts should be able to filter out low-confidence or incomplete rows without asking the extraction system to rerun.

The validator also checks internal completeness against this design. If a projection references `cycle_visibility`, every cycle must have such a judgment. If a decision rule uses `proposal_reengagement`, the event vocabulary must define it. If anonymous aggregate actors are created, they must be counted, scoped, and evidenced. This prevents hidden assumptions from entering the estimator view.

The validator should include cross-document-style consistency checks tailored to merger narratives. A bidder with a final bid but no earlier participation event is a soft flag unless the filing describes a late unsolicited bid. A bidder marked `n/a-admitted` but lacking a post-boundary proposal is a soft flag because the current view will compute `admitted = false`. A bidder with `target-rejected` after a boundary but also a formal bid after that boundary is a hard conflict unless the later event is marked reengagement or out-of-scope.

Cycle-level validation checks reported counts against actor records. If a cycle has a `participation_counts` record showing six IOIs and only three bidder actors with IOI events, the validator requires either three anonymous actors, a documented projection rule that does not create anonymous rows, or a `cycle_visibility = aggregate_only` judgment. The system may choose among these, but it cannot silently lose the count.

Audit reports should include a compact evidence digest. For each exported bidder row, the digest lists actor evidence, informal-bid evidence, formal-bid evidence, boundary evidence, dropout evidence, and scope evidence. The digest is not a substitute for source spans; it is a reviewer navigation aid that makes row-level inspection fast enough to use during corpus expansion.

Reproducibility validation includes diff tests. When a rule change is made, the system compares the new canonical records and views to the prior accepted run and classifies changes as expected, harmless, or breaking. Changes to bid values, boundary dates, dropout labels, cycle counts, or bidder counts are breaking unless the change log explains them and the reference tests are updated.
